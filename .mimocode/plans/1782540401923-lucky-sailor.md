# Phase 3.6 — Tool Invocation

## Goal
AI can call multiple tools in sequence. Workflows reference tools from the registry instead of hardcoding service imports. The LLM can also plan tool call sequences.

---

## Current State
- 11 tools registered in `ToolRegistry` with `execute()` methods
- 9 workflows use hardcoded `from services.xxx import yyy` in each step
- `Tool.execute()` is never called anywhere at runtime
- LLM sees tool descriptions in system prompt but has no invocation path

---

## Design: Tool Invocation Layer

### Core Change
Add a `tool_name` field to `StepSpec`. When present, `WorkflowExecutor` looks up the tool from the registry and calls `tool.execute()` instead of calling `step.fn()` directly.

### Step Types

| Type | Behavior |
|------|----------|
| `fn` (existing) | Calls `step.fn(ctx, db, **params)` directly |
| `tool` (new) | Looks up `tool_registry.get(step.tool_name)` and calls `tool.execute(db=db, **params)` |

### LLM Tool Planning

When the LLM classifies an intent, it can also output a **tool plan** — a sequence of tool calls. The system parses this and executes each tool in order.

**LLM response format** (when tools are involved):
```json
{
  "intent": "analyze_job",
  "tool_plan": [
    {"tool": "profile_get", "params": {}},
    {"tool": "job_analyze", "params": {"job_description": "...", "profile_data": "$profile_get"}},
    {"tool": "applications_list", "params": {}}
  ]
}
```

The `$profile_get` syntax means "use the output of the previous tool as this parameter."

## Step 1: Add tool_name to StepSpec

**File**: `backend/services/workflow.py`

```python
@dataclass
class StepSpec:
    name: str
    step_type: str
    fn: Callable[..., Awaitable[StepResult]] | None = None
    tool_name: str | None = None  # NEW: reference tool by name
    params: dict = field(default_factory=dict)
    optional: bool = False
    param_refs: dict = field(default_factory=dict)  # NEW: {"key": "step_name"} for cross-step data
```

Update `WorkflowExecutor.execute()`:
```python
if step.tool_name:
    tool = tool_registry.get(step.tool_name)
    if not tool:
        return StepResult(success=False, error=f"Tool '{step.tool_name}' not found")
    resolved_params = self._resolve_params(step)
    result_data = await tool.execute(db=self.db, **resolved_params)
    result = StepResult(success=True, data=result_data)
else:
    result = await step.fn(self.context, self.db, **step.params)
```

Add `_resolve_params()` to resolve `$step_name` references:
```python
def _resolve_params(self, step: StepSpec) -> dict:
    params = dict(step.params)
    for key, ref in step.param_refs.items():
        if ref in self.context:
            params[key] = self.context[ref]
    return params
```

## Step 2: Refactor Workflows to Use Tools

**Files**: All 9 workflow files under `services/workflows/`

Each workflow converts hardcoded service calls to tool references:

**Example**: `workflows/analyze_job.py` before:
```python
async def check_profile(ctx, db, **kw):
    from services.profile_service import get_profile
    profile = get_profile(db)
    ...
```

After:
```python
def get_workflow(user_msg, websocket):
    return Workflow(name="analyze_job", steps=[
        StepSpec(name="profile", step_type="tool", tool_name="profile_get"),
        StepSpec(name="job_data", step_type="prepare", fn=prepare_data, params={"user_msg": user_msg}),
        StepSpec(name="analysis", step_type="tool", tool_name="job_analyze",
                 param_refs={"profile_data": "profile"}, params={"job_description": ""}),
        StepSpec(name="save", step_type="db_write", fn=save_application),
        StepSpec(name="respond", step_type="respond", fn=format_response, params={"websocket": websocket}),
    ])
```

Steps that need custom logic (prepare_data, save_application, format_response) keep their `fn` steps. Steps that just call a service become `tool` steps.

## Step 3: Add Tool Planning to LLM Classification

**File**: `backend/routers/chat.py`

When the LLM classifies an intent, also ask it to output a tool plan:

```python
LLM_CLASSIFY_PROMPT = """Classify this user message and create a tool execution plan.

Available tools: {tool_list}

Return ONLY valid JSON:
{
  "intent": "<intent_name>",
  "confidence": 0.0-1.0,
  "tool_plan": [
    {"tool": "tool_name", "params": {"key": "value"}},
    ...
  ]
}
"""
```

The `tool_plan` field is optional — if present, the system can use it to override or validate the workflow.

## Step 4: Add ToolChain Executor

**New concept in `workflow.py`**: A `ToolChain` is a simple sequence of tool calls without the full Workflow/StepSpec machinery:

```python
class ToolChain:
    def __init__(self, steps: list[dict]):
        # steps: [{"tool": "name", "params": {...}, "refs": {"key": "prev_step"}}]
        self.steps = steps

    async def execute(self, db, initial_context=None) -> dict:
        context = initial_context or {}
        for step in self.steps:
            tool = registry.get(step["tool"])
            resolved = self._resolve(step["params"], context)
            result = await tool.execute(db=db, **resolved)
            context[step["tool"]] = result
        return context
```

This allows the LLM to output a tool plan and have it executed directly.

## Step 5: Wire Tool Chain into Chat Handler

**File**: `backend/routers/chat.py`

When the LLM returns a `tool_plan`, execute it via `ToolChain`:

```python
async def _handle_message(...):
    intent = detect_intent(user_msg)
    if intent == "general_chat":
        classified = await classify_intent_with_llm(user_msg)
        if classified != "general_chat":
            intent = classified

    # Check for LLM-generated tool plan
    if hasattr(classified, 'tool_plan') and classified.tool_plan:
        chain = ToolChain(classified.tool_plan)
        result = await chain.execute(db)
        # Format and send result

    # Fall back to workflow
    workflow = get_workflow(intent, user_msg, websocket)
    ...
```
