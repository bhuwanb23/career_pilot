# Phase 3.2 — AI Chat Engine

## Goal
Make the chatbot fully functional: session-aware, conversation memory, proper storage.

---

## Current State
- WebSocket chat at `/ws/chat` works but is **stateless** — each message processed in isolation
- Assistant response saved as hardcoded `"Response sent"` — actual LLM output discarded
- No `session_id` — all messages in one flat table, no conversation isolation
- No conversation context — LLM has zero memory of prior turns
- `prepare_interview` intent detected but has no handler (falls through to general_chat)
- Chat history endpoint returns all messages globally, not per-session

---

## Step 1: Add session_id to ChatMessage model

**File**: `backend/models.py`

- Add `session_id = Column(String(36), index=True, nullable=False)` to `ChatMessage`
- Add `conversation_id` field? No — `session_id` IS the conversation identifier
- Each WebSocket connection gets a UUID4 session_id, sent to client on connect

**Migration**: Drop and recreate table (dev app, no prod data to preserve). Or use `ALTER TABLE` — but since Base.metadata.create_all won't add columns to existing tables, we need to handle this. Simplest: delete the old DB file on startup if schema changed. Actually, just let `create_all` handle it — new column will be added if table doesn't exist, old DB gets wiped by test fixture anyway.

---

## Step 2: Update schemas

**File**: `backend/schemas.py`

- Add `session_id: str` to `ChatMessageResponse`
- Add `ChatSessionResponse` schema: `{ session_id: str, created_at: datetime }`
- Keep `ChatMessageRequest` as-is (just `content: str`)

---

## Step 3: Session management in WebSocket

**File**: `backend/routers/chat.py`

- On WebSocket connect: generate `session_id = str(uuid4())`
- Send `{"type": "session", "session_id": "<uuid>"}` to client immediately after accept
- Pass `session_id` to `_handle_message()` and all DB saves
- Each WebSocket connection = one session (no cross-connection session sharing)

---

## Step 4: Fix assistant response storage

**File**: `backend/routers/chat.py`

- Currently saves `ChatMessage(role="assistant", content="Response sent")` — fix this
- For `general_chat`: collect streamed chunks into a buffer, save the full response text
- For non-streaming intents (`upload_resume`, `analyze_job`, etc.): save the actual text sent to client
- Store `action_type` and `action_data` when applicable

---

## Step 5: Conversation context injection

**File**: `backend/routers/chat.py` (new helper function)

- Add `get_conversation_history(db, session_id, limit=20)` — fetches last N messages for the session
- Format as multi-turn prompt:
  ```
  [Conversation History]
  User: message 1
  Assistant: response 1
  User: message 2
  Assistant: response 2
  ...
  [Current Message]
  User: current message
  ```
- Inject into `general_chat` LLM call instead of just `user_msg + context`
- Keep profile context as system-level info, not part of conversation turns

---

## Step 6: Add prepare_interview handler

**File**: `backend/routers/chat.py`

- The `detect_intent` function returns `"prepare_interview"` but no handler exists
- Add handler: look up the most recent Application, call `generate_prep()`, send summary
- Similar pattern to `analyze_job` handler

---

## Step 7: Update chat history endpoint

**File**: `backend/routers/chat.py`

- Change `GET /api/chat/history` to accept optional `session_id` query param
- If `session_id` provided: filter messages by session
- If not: return all messages (backwards compatible)
- Add `GET /api/chat/sessions` — list all distinct session_ids with last message preview

---
